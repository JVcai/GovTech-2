## 1. Назначение системы

Система решает проблему субъективного и непрозрачного распределения государственных субсидий в АПК Казахстана. Вместо ручного отбора заявок чиновниками — алгоритмическое ранжирование на основе объективных метрик с полным аудитовым следом.

**Ключевые принципы:**

- **Меритократия** — фермеры ранжируются по реальной эффективности, а не по связям
- **Прозрачность** — каждое решение объясняется через SHAP и Gemini-аудитора
- **Законность** — автоматическая проверка соответствия НПА через Compliance Engine
- **Human-in-the-loop** — чиновник может переопределить любое решение ИИ вручную

## 4. Описание модулей

### `data_loader.py` — Модуль 1

**Функция:** `load_and_mock_data(file_path) → pd.DataFrame`

Загружает сырой CSV из системы ИСС (выгрузка МСХ РК). CSV имеет 4 служебные строки в начале (`skiprows=4`). После очистки генерирует **цифровой след** фермера — синтетические данные, имитирующие реальные показатели, которые в обезличенной выгрузке отсутствуют.

| Генерируемая колонка | Логика |
|---|---|
| `years_in_business` | `randint(0, 15)` |
| `past_subsidies` | `Причитающая сумма × uniform(0.5, 3.0)` |
| `produced_volume_kg` | `past_subsidies / Норматив × efficiency` |
| `jobs_created` | `floor(сумма / cost_per_job)`, минимум 1 |
| `tax_debt_amount` | 5% фермеров имеют долг `uniform(100k, 5M)` |

---

### `features.py` — Модуль 2

**Функция:** `generate_features_and_tags(df) → pd.DataFrame`

Четыре последовательных блока обработки:

**Блок 1 — Feature Engineering**

```python
Revenue_Est  = produced_volume_kg × 2500  # ₸/кг (средняя цена)
IFO_ROI      = Revenue_Est / past_subsidies × 100  # в процентах
Sector_Rank  = groupby("Направление водства")["IFO_ROI"].rank(pct=True)
Climate_Weight = CLIMATE_MAPPING[Область]  # нечёткий поиск по подстроке
```

**Блок 2 — Smart Tags** (10 тегов, все векторизованы)

| Тег | Условие |
|---|---|
| Новичок | `years_in_business < 2` |
| Надежный партнер | `tax_debt == 0` AND `IFO_ROI > median` |
| Газель | `years 2–5` AND `Sector_Rank > 0.8` |
| Технологический лидер | `Норматив ≥ p90` по отрасли |
| Якорный инвестор | `Сумма ≥ p95` по региону AND `jobs > median` |
| Экспортный потенциал | `IFO_ROI > median` AND отрасль в экспортных кластерах |
| Эффективный малый бизнес | `Сумма < median` AND `Sector_Rank > 0.7` |
| Социально-значимый | `jobs_created ≥ p80` |
| ⚠️ Нарушитель НПА | `Compliance_Penalty > 0` (добавляется в Блоке 4) |

**Блок 3 — Compliance Engine** (делегируется в `compliance.py`)

**Блок 4 — Scoring Vectors**

```python
Vector_Social       = jobs_created / max_jobs × 100        # [0, 100]
base_reliability    = 60 + (years - 1) / 14 × 40           # [60, 100]
Vector_Reliability  = (base_reliability - Compliance_Penalty).clip(0, 100)
ifo_norm            = IFO_ROI.clip(0, 200) / 2             # → [0, 100]
ML_Score            = Reliability × 0.7 + ifo_norm × 0.3   # [0, 100]
```

# Логика скоринга заявок: почему именно так?

> Документ объясняет бизнес-логику и математику системы оценки фермеров.  
> Целевая аудитория: жюри хакатона, чиновники МСХ РК, новые разработчики.

---

## Главный вопрос: что мы вообще оцениваем?

Государство даёт субсидии не просто так — оно инвестирует бюджетные деньги в сельское хозяйство и ожидает отдачи. Значит, идеальный получатель субсидии — это фермер, который:

1. **Эффективно использует деньги** — берёт субсидию и реально производит продукцию
2. **Соблюдает законы** — не имеет долгов, выполняет нормативы НПА
3. **Создаёт рабочие места** — помогает решать проблему безработицы в аулах
4. **Является надёжным партнёром** — имеет опыт, историю работы

Из этих четырёх задач и строится наш скоринг.

---

## Итоговая формула ML_Score

```
ML_Score = Vector_Reliability × 0.70
         + IFO_ROI_normalized  × 0.30
```

Система намеренно простая и объяснимая. Два компонента — надёжность и экономика.

---

## Компонент 1: Vector_Reliability (вес 70%)

### Почему надёжность важнее экономики?

Потому что государство — не венчурный фонд. Оно не может рисковать бюджетными деньгами на красивые прогнозы. Если фермер нарушает нормативы или имеет долги перед бюджетом — ни о каких субсидиях речи быть не может, сколько бы денег он ни зарабатывал раньше.

Именно поэтому **70% итогового балла** определяется надёжностью — способностью фермера быть законопослушным и ответственным партнёром государства.

### Как считается Vector_Reliability?

```
Vector_Reliability = clip(base_reliability − Compliance_Penalty, 0, 100)
```

#### Шаг 1: base_reliability — базовый балл от стажа

```
base_reliability = 60 + (years_in_business − 1) / 14 × 40
```

| Стаж | Базовый балл | Логика |
|---|---|---|
| 1 год | 60 | Минимум — новичок, риски высокие |
| 5 лет | 71 | Выживший, уже доказал жизнеспособность |
| 10 лет | 86 | Зрелое хозяйство, проверенная модель |
| 15 лет | 100 | Максимум — ветеран отрасли |

**Почему минимум 60, а не 0?**  
Презумпция добросовестности. Мы изначально считаем каждого фермера надёжным — пока не доказано обратное. Даже новичок без истории получает 60 баллов просто за то, что решился зарегистрироваться и подать заявку.

**Почему линейная зависимость, а не ступенчатая?**  
Потому что каждый год работы реально снижает риск банкротства. Статистика говорит, что большинство малых агробизнесов закрываются в первые 2–3 года. Линейный рост справедливо отражает накопление опыта.

#### Шаг 2: Compliance_Penalty — штраф за нарушения НПА

```
Compliance_Penalty = количество_нарушений × 30
```

Каждое нарушение норматива из `newfile.json` стоит **30 баллов штрафа**.

| Нарушений | Штраф | Итог для опытного фермера (стаж 15 лет) |
|---|---|---|
| 0 | 0 | 100 — без изменений |
| 1 | 30 | 100 − 30 = **70** |
| 2 | 60 | 100 − 60 = **40** |
| 3 | 90 | 100 − 90 = **10** |
| 4+ | 120+ | **0** (clip) |

**Почему именно 30 баллов за нарушение?**  
Это калиброванная величина. При одном нарушении фермер всё ещё может пройти (70/100 — приемлемо). При двух нарушениях он попадает в жёлтую зону (40/100 — требует проверки). При трёх и более — красная зона (ниже 10/100 — отклонить). Это соответствует принципу: единичная ошибка простительна, системные нарушения — нет.

**Почему clip(0, 100), а не уходит в минус?**  
Чтобы не создавать ситуации, где штраф влияет на другие метрики за пределами своего компонента. Ноль — это дно, ниже не бывает.

## Компонент 2: IFO_ROI_normalized (вес 30%)

### Что такое IFO_ROI?

```
IFO_ROI = (Revenue_Est / past_subsidies) × 100%

где Revenue_Est = produced_volume_kg × 2500 ₸/кг
```

Это **коэффициент отдачи субсидий** — сколько тенге выручки фермер получил на каждый тенге прошлых субсидий, выраженный в процентах.

| IFO_ROI | Интерпретация |
|---|---|
| < 50% | Критически низкая отдача — деньги расходуются неэффективно |
| 50–100% | Ниже нормы — выручка не покрывает субсидии |
| 100–150% | Норма — хозяйство работает в плюс |
| > 150% | Высокая эффективность — деньги работают хорошо |
| > 200% | Исключительная эффективность |

### Нормализация в [0, 100]

```
IFO_ROI_normalized = clip(IFO_ROI, 0, 200) / 2
```

Мы ограничиваем ROI значением 200% (двукратный возврат) и делим пополам, чтобы получить шкалу 0–100. Это защита от выбросов — агрохолдинг с ROI 1000% не должен "сломать" математику.

### Почему ROI только 30% от итогового балла?

Потому что высокий ROI в прошлом не гарантирует соблюдение законов в будущем. Многие эффективные хозяйства работают в серой зоне, уклоняются от налогов или нарушают нормативы содержания скота. Государство должно стимулировать не просто прибыльность, но и законопослушность.

Если бы ROI весил 70%, система превратилась бы в инструмент поддержки крупных эффективных хозяйств за счёт мелких, но законопослушных фермеров. Это противоречит целям МСХ.

## Target_Score — Ground Truth для обучения модели

```
Target_Score = Sector_Rank × 50
             + Vector_Reliability × 3
             + Vector_Social × 2
```

Это **историческая метрика**, которая нужна только для обучения XGBoost. Она имитирует "идеальное решение" чиновника, который смотрит на три вещи:

### Слагаемое 1: Sector_Rank × 50 (до 50 баллов)

```
Sector_Rank = перцентиль IFO_ROI внутри своей отрасли
```

Это **относительная эффективность** — не абсолютный ROI, а место фермера среди конкурентов в своей отрасли. Лучший птицевод среди птицеводов получает 1.0 × 50 = 50 баллов.

**Почему относительный, а не абсолютный?**  
Разные отрасли несравнимы напрямую. Норма рентабельности для верблюдоводства и для птицефабрики кардинально различаются. Сравниваем фермеров только внутри своей отрасли — это справедливо.

**Почему 50 из 100 возможных?**  
Экономика — главный аргумент, но не единственный. Государство должно поддерживать социально ответственный и законопослушный бизнес, даже если он не самый рентабельный в своей отрасли.

### Слагаемое 2: Vector_Reliability × 3 (до 30 баллов)

```
Vector_Reliability ∈ [0, 10]  →  × 3  →  до 30 баллов
```

(В этой формуле использована старая шкала 0–10, отдельная от финального ML_Score.)

30 из 100 — надёжность важна, но уступает экономике. Идеальная надёжность при нулевой эффективности не помогает отрасли.

### Слагаемое 3: Vector_Social × 2 (до 20 баллов)

```
Vector_Social = jobs_created / max_jobs_in_dataset × 100
             → ∈ [0, 10]  →  × 2  →  до 20 баллов
```

20 из 100 — государство субсидирует не только производство, но и занятость. Фермер, который нанимает людей, решает проблему безработицы в ауле. Это стратегическая цель МСХ, поэтому социальный компонент обязателен, но не доминирует.

### Итого 100 баллов

| Компонент | Макс. баллов | Смысл |
|---|---|---|
| Sector_Rank × 50 | 50 | Экономическая эффективность в своей нише |
| Vector_Reliability × 3 | 30 | Надёжность и законопослушность |
| Vector_Social × 2 | 20 | Социальная ответственность |
| **Итого** | **100** | |

Эта формула — **язык чиновников**: производи больше всех в своей отрасли (50 баллов), плати налоги и соблюдай нормативы (30 баллов), давай работу людям (20 баллов).

## Как связаны Target_Score и ML_Score?

```
Target_Score  →  обучает XGBoost  →  предсказывает  →  ML_Score
   (Ground Truth,              (на новых данных,
    исторические данные)        без знания Target)
```

**Target_Score** — это то, как должна была бы выглядеть идеальная оценка на исторических данных.  
**ML_Score** — это то, что XGBoost предсказывает для новых заявок на основе 5 признаков (`Причитающая сумма`, `Норматив`, `years_in_business`, `jobs_created`, `Climate_Weight`).

После прогона через `predict_and_explain()` **ML_Score пересчитывается** по формуле из features.py:

```
ML_Score = Vector_Reliability × 0.7 + IFO_ROI_normalized × 0.3
```

Это более прозрачная и объяснимая формула — чиновник может проверить её вручную по двум числам. XGBoost-версия используется для обучения SHAP-объяснений.

## Compliance_Penalty: почему нарушение НПА так дорого стоит?

Вернёмся к примеру: фермер с 15-летним стажем (base=100) нарушил один норматив.

```
Vector_Reliability = 100 − 30 = 70
ML_Score = 70 × 0.7 + ... = 49 + ...
```

Даже у лучшего фермера одно нарушение "съедает" 30% веса надёжности, что немедленно снижает итоговый балл примерно на **21 пункт** (30 × 0.7 = 21).

**Это намеренно жёстко.** Логика следующая:

> Если фермер знает, что нарушение норматива стоит ему всего 2–3 балла из 100 — он проигнорирует его. Если нарушение стоит 21 балл и переводит его из "APPROVED" в "REJECTED" — он дважды подумает перед нарушением.

Compliance Engine должен не просто фиксировать нарушения, а **реально влиять на деньги**. Именно поэтому компонент надёжности занимает 70% финального балла.

## Climate_Weight: почему Мангистауская получает 1.2, а Алматинская 0.8?

```
Climate_Weight ∈ {0.8, 1.0, 1.2}
```

Это поправочный коэффициент, который **НЕ влияет напрямую на ML_Score**, но используется как признак в XGBoost.

Логика: фермеры в засушливых регионах (Мангистауская, Кызылординская) работают в объективно более тяжёлых условиях. При равном ROI они должны получать приоритет — их эффективность достигнута вопреки климату, а не благодаря ему.

Коэффициент 1.2 говорит XGBoost: "при прочих равных — эти фермеры ценнее". Это механизм **региональной справедливости** в системе.

### `compliance.py` — Rules Engine

**Функция:** `apply_compliance_engine(df, rules_path="newfile.json") → pd.DataFrame`

Читает `newfile.json` и проверяет каждого фермера по всем правилам. Поддерживает два формата правил:

| `water`, `cert`, `rate` | `U(0, 1)` — индексы и доли |

### `ml_engine.py` — Модуль 3

**Функция-обёртка:** `train_and_explain(df) → pd.DataFrame`

Единая точка входа для `app.py`. Последовательно вызывает:

1. **`build_target(df)`** — формирует Ground Truth:
   ```
   Target_Score = Sector_Rank × 50 + Vector_Reliability × 3 + Vector_Social × 2
   ```

2. **`clip_outliers(df)`** — обрезает выбросы по `p1/p99` перед обучением. Защита от **Outlier Squeeze** — ситуации, когда один фермер с аномальным значением (`years=999`) сжимает всех остальных в один лист дерева.

3. **`train_model(df)`** — обучает `XGBRegressor`:
   - Признаки: `Причитающая сумма`, `Норматив`, `years_in_business`, `jobs_created`, `Climate_Weight`
   - Параметры: `n_estimators=100`, `max_depth=5`, `learning_rate=0.1`
   - Split: 80/20, `random_state=42`
   - Логирует MAE и R² на тест-выборке

4. **`predict_and_explain(df)`** — прогноз + SHAP:
   - SHAP `TreeExplainer` вычисляет матрицу `(n_rows × n_features)`
   - Защита от shape mismatch (разные версии библиотеки shap возвращают `list` или `ndarray`)
   - `SHAP_Report` — текст на русском: главный драйвер + сдерживающий фактор

### `allocator.py` — Модуль 4

**Функция:** `run_waterfall_allocation(df, total_budget, max_cap_pct=0.15, required_tags=None) → pd.DataFrame`

Четыре последовательных этапа:

```
Этап 1: GATING FILTER
  tax_debt_amount > 0  →  Status = REJECTED_COMPLIANCE
  
Этап 2: POLICY FILTER
  если required_tags задан:
    нет пересечения тегов  →  Status = REJECTED_POLICY
    
Этап 3: MERITOCRATIC RANKING
  сортировка по ML_Score ↓ (лучшие идут первыми)
  
Этап 4: WATERFALL + АНТИМОНОПОЛЬНЫЙ КАП
  max_allowed = total_budget × max_cap_pct
  
  для каждого фермера сверху вниз:
    effective_request = min(Причитающая сумма, max_allowed)
    
    если budget_left >= effective_request:
      → Status = APPROVED,  Allocated = effective_request
      
    elif budget_left > 0:
      → Status = PARTIAL,   Allocated = budget_left
      
    else:
      → Status = REJECTED_NO_FUNDS, Allocated = 0
```

## 6. Скоринговая модель

### Формула ML_Score (финальная, v5.x)

```
ML_Score = Vector_Reliability × 0.70
         + IFO_ROI_normalized  × 0.30

где:
  Vector_Reliability = clip(base_reliability − Compliance_Penalty, 0, 100)
  base_reliability   = 60 + (years_in_business − 1) / 14 × 40   ∈ [60, 100]
  IFO_ROI_normalized = clip(IFO_ROI, 0, 200) / 2                ∈ [0, 100]
```

**Интерпретация весов:**
- **70% — правовая надёжность.** Государство не субсидирует нарушителей.
- **30% — экономика.** Эффективность использования прошлых субсидий.

### Пример расчёта

| Параметр | Значение |
|---|---|
| Стаж | 8 лет → `base = 60 + 7/14 × 40 = 80` |
| Compliance_Penalty | 30 (одно нарушение) |
| Vector_Reliability | `80 − 30 = 50` |
| IFO_ROI | 120% → `norm = 60` |
| **ML_Score** | **`50 × 0.7 + 60 × 0.3 = 35 + 18 = 53`** |

## 8. Waterfall Allocator

### Статусы и их значение

| Статус | Значение |
|---|---|
| `APPROVED` | Полная сумма выделена |
| `PARTIAL` | Выделены остатки бюджета (бюджет закончился на этом фермере) |
| `REJECTED_NO_FUNDS` | Бюджет исчерпан до этой заявки |
| `REJECTED_COMPLIANCE` | Налоговая задолженность (hard block) |
| `REJECTED_POLICY` | Не соответствует приоритетным тегам |
| `MANUAL_APPROVED` | Одобрен чиновником вручную |
| `MANUAL_REJECTED` | Отклонён чиновником вручную |

### Human-in-the-Loop

При ручном переопределении бюджет пересчитывается через `manual_budget_delta` — дельта накапливается в `session_state` без перезапуска всего Waterfall. Это позволяет делать точечные правки, не инвалидируя весь расчёт.

## 12. FAQ и известные особенности

**Q: Почему все ML_Score похожи (~95)?**

A: Это Outlier Squeeze. Проверь что `clip_outliers()` вызывается в `train_model()`. Один фермер с аномальным значением (например, `years_in_business=999`) сжимает всех остальных в один лист дерева XGBoost.

**Q: Compliance_Logs пустой у всех фермеров**

A: Проверь порядок вызовов в `generate_features_and_tags()`. `apply_compliance_engine()` должен вызываться **до** расчёта `Vector_Reliability`, иначе штрафы не попадают в вектор.

**Q: Ошибка `SHAP shape mismatch`**

A: Разные версии библиотеки `shap` возвращают разный тип из `explainer.shap_values()`. В `ml_engine.py` есть проверка `isinstance(shap_raw, list)` — убедись что она на месте.

**Q: `_GEMINI_AVAILABLE is not defined`**

A: Переменная называется `GEMINI_AVAILABLE` (без подчёркивания). Проверь все места использования.

**Q: `session_state key cannot be modified after widget instantiation`**

A: Ключ `session_state` совпадает с `key` виджета. Переименуй ключ кнопки: `key=f"btn_audit_{row_id}"` вместо `key=f"audit_{row_id}"`.

**Q: PDF не читается**

A: Установи `pdfplumber`:
```bash
pip install pdfplumber
```
Если PDF сканированный (картинка) — текст не извлечётся, система сгенерирует синтетические данные.

**Q: Как добавить новый норматив НПА?**

A: Добавь запись в `newfile.json`. Код менять не нужно — Rules Engine подхватит автоматически.
